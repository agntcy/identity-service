What this demo shows

A cross-domain AI agent remediation scenario where Sarah (an engineer at Org A) asks OpenCode (her Org A AI agent) to fix a CVE found in a repo owned by Org B. Because Org B has its own IdP and access control, OpenCode cannot act directly — it must assert Sarah's delegation cross-domain using the ID-JAG protocol.

Step-by-step walkthrough

Phase A — Org A setup (steps 1–2)

Sarah signs in via OIDC password grant using the opencode-agent client in Org A's Keycloak realm. OpenCode then runs a mock CVE scan and decides the fix requires cross-domain action.

AGNTCY Directory — per-turn record & agent discovery (steps 3–4)

OpenCode pushes an OASF record to the AGNTCY Directory Node (dir-apiserver:8888, gRPC). The directory returns a CID — a content-addressed, immutable identifier that proves this remediation turn happened. OpenCode then searches the directory by agent name to discover the triage-agent's endpoint.

# Push turn record (gRPC bidirectional stream)
stub.Push([Record(data={
    "name": "opencode-agent", "schema_version": "1.1.0",
    "annotations": {"cve": "CVE-2024-12345", "repo": "…"}
})])
→ RecordRef(cid="baeare…")   # immutable, content-addressed

# Discover triage-agent
stub.SearchRecords(SearchRecordsRequest(queries=[RecordQuery(
    type=RECORD_QUERY_TYPE_NAME, value="triage-agent"
)]))
→ [SearchRecordsResponse(record=...)]

CIMD — real local trust authority (steps 5–6)

identity-node's actual REST API has no "register app / issue badge" concept — it needs a self-issued proof JWT to mint or resolve an id. At startup, identity-node-init registers org-a as a local trust authority: it creates an RSA-2048 key in Vault's transit engine (the private key never leaves Vault), reads back the public key, and self-signs a proof JWT (iss=agntcy:org-a, a sub_jwk claim carrying the public key) via Vault's /transit/sign API to register the issuer. Every subsequent call — including the ones below, from the webapp itself — signs a fresh proof JWT the same way to mint or resolve an id for triage-agent under org-a's authority.

POST /v1alpha1/id/generate
{
  "issuer": {"organization": "org-a", "commonName": "org-a"},
  "proof": {"type": "JWT", "proofValue": "<Vault-signed JWT, iss=agntcy:org-a, sub=triage-agent>"}
}
→ {"resolverMetadata": {"id": "AGNTCY-triage-agent", "verificationMethod": [...]}}

POST /v1alpha1/id/resolve
{"id": "AGNTCY-triage-agent"}
→ {"resolverMetadata": {...}}   # same ResolverMetadata, read back

Cross-domain token exchange (steps 7–10)

OpenCode mints an ID-JAG assertion at idjag-issuer: a short-lived RS256 JWT with sub=sarah@org-a.example, act_chain=[opencode-agent], aud=KC-B org-b realm, and scope=triage:create. Org B's Keycloak validates the assertion against the issuer's JWKS, maps Sarah's identity, and issues a scoped access token.

POST /mint   # idjag-issuer
{
  "sub": "sarah@org-a.example",
  "aud": "http://keycloak-b:8080/realms/org-b",
  "client_id": "triage-agent",
  "act_chain": ["opencode-agent"],
  "scope": "openid triage:create"
}
→ {"assertion": "eyJ…"}

POST /realms/org-b/protocol/openid-connect/token
  grant_type = urn:ietf:params:oauth:grant-type:jwt-bearer
  assertion  = eyJ…
  client_id  = triage-agent
  scope      = triage:create
→ {"access_token": "eyJ…", "scope": "triage:create"}

Org B — Triage agent & bounded sub-agent (steps 11+)

OpenCode calls POST /api/ticket on the Triage agent with the scoped token. Triage runs an OPA policy check, plans the fix, then mints a narrowed sub-badge for Sub-Agent: caps ⊆ parent — only gitea:write gitea:pr (no triage:create), with a nested act-chain: Sarah → OpenCode → Triage → Sub-Agent. Sub-Agent redeems the sub-badge at Keycloak B, pushes the fix file to a feature branch, and opens a PR through the Gitea gateway.

Architecture

Keycloak 26.7 feature flags required: --features=identity-assertion-jwt,token-exchange-delegation,parameterized-scopes
These enable the ID-JAG receiver (jwt-bearer grant), RFC 8693 delegation scope, and parameterized scopes for narrow tokens.
