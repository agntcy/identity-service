What is Cross-App Access (ID-JAG)?

Cross-App Access (XAA) lets one application (the Requesting App) obtain access to another (the Receiving App) on behalf of a signed-in user — without the Receiving App ever seeing the user's password, and without the user re-consenting for every app pair.

It is built on two OAuth 2.0 building blocks:

  • ID-JAG (Identity Assertion JWT Authorization Grant) — a central IdP mints a short-lived, signed assertion that says "this user authorizes this app to act". draft-ietf-oauth-identity-assertion-authz-grant.
  • Token exchange (RFC 8693) — a service swaps one token for another, optionally downscoping it to just what the next call needs.
Keycloak's role today: Keycloak 26.7 implements the ID-JAG Receiver only — it accepts an externally-signed assertion at its token endpoint and issues a local access token. It cannot yet mint assertions, so this demo uses a small idjag-issuer service as the stand-in enterprise IdP.

How Keycloak is configured

1. Experimental feature flags

Keycloak is started with the flags from the ID-JAG guide:

start-dev --import-realm \
  --features=identity-assertion-jwt,token-exchange-delegation,parameterized-scopes
  • identity-assertion-jwt — enables the ID-JAG receiver (the jwt-bearer grant). Needs Keycloak 26.7+.
  • token-exchange-delegation — enables the delegation scope for RFC 8693 token exchange.
  • parameterized-scopes — dependency of the delegation feature.

2. The Receiving App client (the ID-JAG receiver)

In keycloak/cncf-demo-realm.json, receiving-app is a confidential client with the ID-JAG grant enabled:

{
  "clientId": "receiving-app",
  "serviceAccountsEnabled": true,
  "attributes": {
    "oauth2.jwt.authorization.grant.enabled": "true",
    "oauth2.jwt.authorization.grant.idp": "id-jag"
  }
}

The ...grant.idp attribute points at the identity provider Keycloak trusts to have signed incoming assertions.

3. The id-jag identity provider (the trust anchor)

An OIDC identity provider named id-jag tells Keycloak where to fetch the issuer's public keys (JWKS) so it can verify assertion signatures, and jwtAuthorizationGrantEnabled=true allows it to be used for the grant. In this demo it points at idjag-issuer; swap the issuer/JWKS URLs for your real enterprise IdP.

4. Narrow scopes (registered by kc-init)

The gitea:read / gitea:write client scopes are registered with kcadm after the realm import and assigned as optional scopes on receiving-app — so they are only granted when explicitly requested:

kcadm.sh create client-scopes -r cncf-demo \
  -s name=gitea:read -s protocol=openid-connect \
  -s 'attributes."include.in.token.scope"=true'
# then assign as an OPTIONAL scope on receiving-app
Hand-authoring clientScopes in a realm import replaces Keycloak's built-in scopes (profile, email, …), so the demo registers the gitea scopes via kcadm after import instead — keeping the standard scopes intact.

5. The exchange call

The Receiving App presents the assertion to Keycloak's token endpoint. Keycloak verifies the signature (against the issuer JWKS), the audience and client, maps sub to the local user via the federated identity link, and returns a downscoped access token:

POST /realms/cncf-demo/protocol/openid-connect/token
  grant_type = urn:ietf:params:oauth:grant-type:jwt-bearer
  assertion  = <signed ID-JAG JWT>
  client_id  = receiving-app
  scope      = openid gitea:read        # request only what you need

More sequences

A. One-time trust setup today

Keycloak learns to trust the issuer's signatures via the id-jag IdP's JWKS URL; kc-init adds the narrow scopes.

B. The exchange & narrow-scoping enforcement live

The token is only as powerful as the scope requested; the gateway (an OAuth resource server) refuses out-of-scope operations.

Identity node & Verifiable Credentials

The stack also runs the AGNTCY identity node (REST :4003, gRPC :4004). It is a registry and resolver for agent/app identities and their Verifiable Credentials (VCs) — AGNTCY calls these Badges (e.g. AgentBadge, McpBadge), which are W3C VCs carried in a JOSE envelope.

Node endpoint (/v1alpha1)Purpose
POST /issuer/registerRegister a tenant issuer + its public JWK
POST /id/generateMint a resolvable identity (ResolverMetadata) from an OIDC proof
POST /vc/publishStore a signed VC (validates the OIDC proof against the subject)
POST /vc/verifyCryptographically verify a VC
GET /vc/{id}/.well-known/vcs.jsonPublic resolution of an identity's VCs

VC signing happens in the Identity Service BFF (with the tenant key from Vault); the node publishes / resolves / verifies. The CNCF stack currently runs the node but not the BFF.

C. Issuing a VC (Badge) with the identity node today

A Badge binds an app/agent's resolver_metadata_id to signed metadata (agent card / MCP definition), publicly resolvable via well-known.

D. VC-backed ID-JAG — how VCs plug into this flow proposed

The gateway (or Keycloak, via the DCR vc+jwt / Client-ID-Metadata proposal) verifies the agent's VC through the identity node before honoring the token — adding cryptographic proof of the calling agent's identity on top of the user-delegated access.

Why this matters: ID-JAG proves the user authorized this app. The VC proves this app/agent is who it claims to be (issued and signed by a trusted issuer, resolvable and revocable). Together you get delegated access plus verifiable agent identity — least privilege with provenance.
Wiring it live (next step): issuing a VC needs a signer. Two options: (1) run the Identity Service BFF, which signs Badges with the tenant key and publishes them to the node; or (2) have idjag-issuer double as a VC issuer — it already holds an RSA key + JWKS, so it can register with the node, sign a Badge for requesting-app, publish it, and add the resolver_metadata_id as a claim in the ID-JAG assertion, with the gateway calling /vc/verify before allowing the Gitea call.
